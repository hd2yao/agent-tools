# Codex 工作台产品化与公开发行设计

日期：2026-07-20

状态：已批准，待实施

## 背景

Codex 工作台已经完成操作日志 V1.4 和 Profile Switcher 核心能力整合，能够在一个原生 macOS App 中展示账号、额度、重置卡、运行状态、项目与工具数据。当前仍有四类产品化缺口：

- 侧栏保留“即将推出”的重复入口，模块边界不清楚；
- 当前账号缺少独立的一键重启入口，“高级诊断”也没有形成可操作的诊断能力；
- 账号入口假设存在 `~/.codex-profiles`，无法自然服务只有 `~/.codex` 单账号的普通用户；
- 安装仍依赖本机脚本、ad-hoc 签名和外部 Python，不能以 DMG 交付给其他用户。

本阶段把工作台收敛成唯一日常 App，并完成最小公开发行闭环。旧 Profile Switcher 保留为冷备，但不再作为日常入口。

## 已批准的产品决策

- 最终日常形态只有一个 Codex 工作台 App。
- 旧 `Codex Profile Switcher.app` 先做版本化冷备；迁移或验收失败时可退出工作台后回退。
- 菜单栏继续优先显示 Codex 当前实际登录账号的额度与工作状态。
- 空闲状态可一键重启当前 Codex；存在运行中或待接手任务时必须先确认风险。
- 普通单账号用户直接识别现有 `~/.codex`，不迁移认证文件、不创建 profile、不建立软链。
- 已经存在 `~/.codex-profiles` 的用户继续保留现有账号切换、额度、重置卡和安全重启能力。
- 首发不提供新增账号、导入账号、删除账号或单账号转多账号流程。
- 首发只通过 GitHub Releases 发布 DMG；不进入 Mac App Store，不提供 Homebrew，不做自动检查或自动更新。
- 只支持 Apple Silicon（M 系列芯片）和项目现有的 macOS 13 及以上目标，不构建 Intel 或 Universal 版本。
- 公开发行包必须自包含，用户不需要安装 Python、Homebrew、Xcode 或运行终端命令。

## 目标

1. 清理工作台信息架构，合并重复模块并移除静态“即将推出”占位区。
2. 恢复当前账号的一键安全重启，并把高级诊断变成真实、可操作的诊断与修复入口。
3. 兼容普通单账号和既有多 Profile 两种本机状态，不复制第二套账号状态源。
4. 产出经过 Developer ID 签名、Apple 公证且可在干净 Apple Silicon Mac 上直接安装的 DMG。
5. 通过 GitHub Releases 手动发布版本；升级由用户重新下载 DMG 完成。

## 非目标

- 不做 Intel / Universal 构建。
- 不进入 Mac App Store、TestFlight 或 Homebrew Cask。
- 不集成 Sparkle，不自动联网检查或下载更新。
- 不实现新增账号、导入账号、删除账号、登录新账号或单账号转多账号向导。
- 不改变现有 profile 目录、认证文件格式或安全切换事务。
- 不删除旧 Profile Switcher 源码、CLI、冷备 App 或恢复记录。
- 不重做已经完成的操作日志 V1.4。
- 不复制 Codex 对话 UI，不把工作台变成第二个聊天客户端。
- 不新增遥测、分析上报或云端账号服务。

## 总体架构

```text
Codex 工作台 App（唯一日常宿主）
├── 菜单栏：当前账号额度 + Codex 工作状态 + 快捷操作
├── 概览
├── 操作日志
├── 账号管理
│   ├── 本机单账号适配
│   ├── 既有 Profiles 适配
│   ├── 当前账号重启
│   └── 诊断与修复
├── 项目与任务
├── 工具与自动化
└── 内置账号后端 / 辅助程序

Codex Profile Switcher 冷备
└── 保留安装与恢复说明，不与工作台并行运行
```

工作台继续只有一个账号协调层。主窗口、菜单栏、重启、诊断和操作日志必须消费同一份账号与运行状态快照，不能分别读取认证文件并得出不同结论。

## 信息架构

侧栏收敛为五个稳定入口：

1. **概览**：跨模块摘要、异常提醒和最近重要操作。
2. **操作日志**：保留现有时间轴与详情能力。
3. **账号管理**：当前账号、额度、重置卡、既有账号切换、当前账号重启和诊断。
4. **项目与任务**：合并“项目分析”与“任务与线程”；按项目进入任务 / 对话列表与上下文状态，不复制聊天正文体验。
5. **工具与自动化**：合并“工具 / Skill”与“Hook 与自动化”；使用分段或页内分类展示工具、Skill、Hook 和 Automation。

“上下文健康”不再成为独立侧栏入口：单任务健康进入“项目与任务”详情，跨任务风险进入概览提醒。删除整个“即将推出”静态分组。

## 账号兼容模型

### 本机单账号模式

识别条件：存在可用的 `~/.codex`，但不存在可管理的 profile 集合。

- 将当前状态展示为“本机当前账号”，账号身份以本机 Codex App Server 返回值为准。
- 展示当前登录账号、套餐、官方额度、重置时间、重置卡和 Codex 工作状态。
- 不移动或复制 `auth.json`，不创建 `~/.codex-profiles`，不建立 profile 软链。
- App Server 读取只允许在权限为 0700 的短生命周期临时 home 中创建指向原 `auth.json` 的符号链接；认证正文始终只有原文件一份。子进程必须通过系统沙盒禁止写入原默认 home 与认证实际目标，数据库、Skills 和日志只允许落入临时 home；沙盒不可用时失败关闭。
- 不显示“添加账号”；没有其他账号时也不显示空的切换列表。
- 无法读取账号时显示诊断原因，不把目录名或历史任务推断当成当前账号。

### 既有 Profiles 模式

识别条件：已有 Profile Switcher 管理的 profile 目录和桥接状态。

- 继续由现有成熟账号后端作为认证、额度、重置卡和切换的唯一权威来源。
- 保留已有账号列表以及“切换并重启 Codex”。
- 不改变 profile 名称、路径、认证格式或桥接方式。
- 工作台与旧冷备不得同时执行切换或自动重置。

### 模式选择

每次刷新先判断既有 Profiles 是否可用；可用时进入既有 Profiles 模式，否则进入本机单账号模式。模式变化必须记录为诊断状态，但不能自动迁移数据。

## 当前账号重启

账号管理页和菜单栏都提供“重启当前 Codex”：

1. 读取统一运行状态。
2. 若为空闲，直接复用既有安全退出、等待、重新启动和启动验证流程。
3. 若为运行中或待接手，显示明确风险确认；取消时不做任何操作。
4. 若状态未知，按高风险处理并要求确认。
5. 重启完成后重新读取真实状态；只有重新启动并可识别时才提示成功。
6. 失败时显示失败阶段与手动恢复建议，不改变当前账号路由，不伪报成功。

重启不执行账号切换，也不消费重置卡。开始、取消、成功和失败进入操作日志，但不记录凭据内容。

## 诊断与修复

原“高级诊断”拆成两层：

- 账号页内保留默认折叠的“账号来源说明”，解释当前账号、任务推断和统计归因的区别。
- 提供可点击的“诊断与修复”入口，打开独立 sheet 或页面并执行真实检查。

首发诊断项：

- 是否找到官方 Codex bundle id `com.openai.codex`；
- 是否同时存在 `ChatGPT.app` 与 `Codex.app` 的重复安装 / 注册；
- 当前实际运行的 App 路径、版本和进程状态；
- `~/.codex` 是否可读、账号模式及最近成功刷新时间；
- 既有 Profiles 数量、active profile 与桥接健康；
- 内置账号后端 / 辅助程序是否存在、可执行且版本匹配；
- 最近一次刷新、重启或切换的失败阶段。

安全操作限定为刷新诊断、打开官方 Codex、重启当前 Codex、在 Finder 显示相关 App / 目录和复制脱敏诊断摘要。首发不提供删除 App、修改认证文件、修复软链或新增账号的一键操作。

## 菜单栏与页面行为

菜单栏状态项仍只突出当前实际登录账号的首要额度和 Codex 工作状态。展开面板保留当前账号摘要、两档额度、重置卡摘要、已有账号快速切换、重启当前 Codex、打开工作台、打开 Codex、刷新和退出。

账号管理页按以下顺序展示：当前账号与状态、官方额度、重置卡、账号用量、已有其他账号、账号来源说明、诊断与修复。单账号模式下自然省略“其他账号”。所有页面继续遵守 `DESIGN.md` 的 Calm Operations Console，不恢复旧工具的独立视觉体系。

## 发行设计

### 用户安装路径

1. 用户从 GitHub Releases 下载 `Codex-Workbench-v<版本>-arm64.dmg`。
2. 打开 DMG，把 `Codex 工作台.app` 拖入 `/Applications`。
3. 首次打开通过 macOS Gatekeeper 检查。
4. 工作台自动识别本机 Codex 和账号模式并进入可用页面。

### 发行产物

- 只构建 `arm64` Release 二进制。
- 主 App、登录辅助程序、内置账号后端及所有嵌套可执行文件使用 Developer ID 签名并启用 Hardened Runtime。
- 创建 DMG 后提交 Apple notarization，并把票据 staple 到发行容器。
- Release 同时提供 SHA-256、版本说明、最低系统版本、仅支持 Apple Silicon 的明确提示和手动升级说明。
- GitHub Release 上传由显式发布脚本完成；创建 Release、上传或覆盖外部资产仍需发布者主动执行。

### 手动升级

- App 不启动后台更新器，也不自动请求 GitHub API。
- 用户获取新版 DMG 后，用新版 App 覆盖旧版；本地账本、设置和账号目录不随 App 覆盖删除。
- “关于”或帮助区域可以提供用户主动点击的 GitHub Releases 链接，但不自动检查版本。

### 自包含运行时

公开安装包不得依赖外部 `python3`。现有成熟 Python 账号逻辑保持单一权威实现，但必须封装为 App 内经过签名的自包含辅助产物；Swift 只调用包内固定路径。若自包含方案无法满足 Hardened Runtime、公证、体积和干净机验证，必须回到实现方案评审，不得悄悄恢复系统 Python 依赖，也不得直接重写认证逻辑。

## 数据、安全与隐私

- 不输出、上传或持久化 token、Cookie、`auth.json` 内容和完整重置卡原始 ID。
- 账号读取继续通过本机 Codex App Server 与本地文件完成。
- 不新增遥测、后台更新请求或云端同步。
- 诊断摘要默认脱敏，只包含路径类别、版本、状态、错误类型和时间。
- 安装与升级不修改用户账号目录；既有 profile 写操作仍只允许成熟后端执行。

## 错误处理

- 找不到 Codex：工作台仍可打开，显示安装提示和诊断入口。
- 单账号状态不可读：保留最近成功快照并标记暂存；没有快照时显示 `--` 和原因。
- 内置后端失败：账号模块局部降级，操作日志和其他模块继续可用。
- 重启失败：显示退出、等待、启动或验证中的具体失败阶段。
- 重复官方 App：不自动删除或改名；展示当前实际运行与 Launch Services 解析结果。
- 公证或 Gatekeeper 验证失败：不得创建 GitHub Release。

## 测试与验收

### 自动化

- 单账号、既有 Profiles、无 Codex、不可读和模式变化的纯逻辑测试。
- 当前账号重启的空闲、运行中、待接手、未知、取消和失败阶段测试。
- 诊断项、脱敏摘要和安全动作契约测试。
- 侧栏模块、路由和上下文健康归属测试。
- 包内后端定位测试，明确拒绝外部 Python 回退。
- arm64、签名层级、Hardened Runtime、DMG、公证票据和 SHA-256 验证脚本。

### 真实本机验收

- 当前已有 Profiles 机器识别既有账号并完成双向切换后恢复初始账号。
- 空闲状态重启当前 Codex；运行中状态只验证风险确认，不中断真实任务。
- 诊断页面能够识别当前重复的 `ChatGPT.app` / `Codex.app` 状态。
- 旧 Profile Switcher 保持冷备且不与工作台并行运行。

### 干净机验收

在没有仓库、Homebrew、Python 或既有 Profiles 的 Apple Silicon Mac / 干净测试用户中：

- 下载 DMG 并拖入 Applications 后可正常打开；
- Gatekeeper 接受签名与公证；
- 只有一个现有 Codex 登录账号时能直接显示当前账号、额度和运行状态；
- 不创建 profile 或软链，不要求终端命令；
- 退出、重开和用新版覆盖安装后仍可使用。

## 迁移与回滚

1. 记录旧 Profile Switcher 稳定 App、源码提交、二进制 hash 和账号后端版本。
2. 建立版本化冷备及恢复说明；冷备默认退出且不配置为日常启动项。
3. 逐批实现信息架构、单账号适配、重启、诊断和发行链路。
4. 每批保持既有 Profiles 数据格式不变并运行回归。
5. 任一账号切换、认证、重置卡或公开安装门禁失败时，停止发行并恢复到上一已验收工作台；必要时退出工作台后启用冷备。
6. 本阶段不删除冷备。正式退役仍需用户另行批准。

## 完成定义

- 日常只需要启动 Codex 工作台；旧 Profile Switcher 仅为冷备。
- 侧栏不存在重复的“即将推出”入口。
- 当前账号可安全重启，高风险状态有确认。
- 诊断入口可点击且返回真实、脱敏、可行动的结果。
- 单账号用户无需迁移即可看到当前登录账号额度；既有 Profiles 用户能力不缩水。
- GitHub Release 中存在仅面向 Apple Silicon、签名公证通过、无需外部依赖的 DMG，并有干净机安装证据。
- App 不自动检查或安装更新。
