# Codex 工作台与账号工具整合设计

日期：2026-07-17

状态：已批准，待实施

## 背景

当前本机同时运行两个原生 macOS App：

- `Codex 观测站.app`：提供概览、操作日志、账号摘要和工作台菜单栏。
- `Codex Profile Switcher.app`：提供当前账号额度、Codex 对话运行状态、账号切换、重置卡、通知、自动重置和用量分析。

观测站已经把 Profile Switcher 的 Python 后端打包为账号适配器，但只解码和展示部分字段。运行灯、重置卡逐张详情、到期通知、自动使用重置卡、用量趋势、项目排行和工具 / Skill 排行仍由独立 Profile Switcher App 承载。两个 App 同时运行会产生重复入口，也可能造成重复刷新、通知或自动重置竞争。

本次整合采用“模块化迁移、保留成熟账号引擎”的方案：工作台成为唯一日常产品入口，现有 Python 账号引擎继续作为单一权威状态源，旧 Profile Switcher App 在迁移期只作为冷备保留。

## 目标

1. `Codex 观测站` 成为主项目、主 App 和所有 Codex 小工具的容器。
2. 日常只运行一个工作台 App，不再同时运行两个菜单栏进程。
3. 完整迁移 Profile Switcher 已有的账号、额度、重置卡、切换、运行状态、通知和自动重置能力，不做能力缩水。
4. 菜单栏继续优先显示当前登录账号的额度和 Codex 真实工作状态。
5. 账号详细信息进入工作台“账号管理”页面，并遵循现有 Design Lock。
6. 用量、项目和工具 / Skill 信息按业务归属拆成工作台模块。
7. 迁移经过真实安装、双向账号切换和视觉验收后，再单独进入旧 App 退役阶段。

## 非目标

- 本阶段不删除旧 Profile Switcher App、源码或 CLI。
- 本阶段不重写成熟的 Python 认证、账号桥接和安全切换逻辑。
- 本阶段不扩展“任务与线程、Hook 与自动化、上下文健康”等未来模块。
- 不为了测试而故意消耗真实重置卡。
- 不新增遥测、外部网络服务或第三方 Swift 依赖。

## 已确认的产品决策

- 旧 Profile Switcher App 在迁移期作为冷备：保留安装，但默认退出；只有工作台切换失效时才在退出工作台后手动启用。
- 最终形态只有一个工作台 App；CLI 兼容入口可以作为内部后端保留。
- “账号管理”承载账号身份、额度、重置卡、切换和账号用量。
- “项目分析”承载项目排行；“工具与 Skill”承载调用排行与相关统计；概览只展示摘要。
- 菜单栏额度始终绑定 Codex 当前实际登录账号，不受历史任务账号推断或统计归因覆盖。
- 手动打开工作台时显示主窗口；登录 Mac 自动启动时只驻留菜单栏。

## 总体架构

```text
Codex 工作台 App（唯一日常宿主）
├── 菜单栏与主窗口生命周期
├── 操作日志
├── 账号协调服务
│   ├── 现有 Python 账号引擎（唯一账号状态源）
│   ├── Codex 运行状态监听
│   ├── 重置卡通知
│   └── 自动重置状态机
├── 项目分析
└── 工具与 Skill

旧 Profile Switcher App
└── 迁移期冷备；不得与工作台并行运行
```

### 单一账号状态源

现有 `codex_profile.py` 和 `codex_profile_dashboard.py` 继续负责：

- profile 创建、同步和状态读取；
- 默认 `~/.codex/auth.json` 账号桥接；
- Codex 认证文件刷新归档与冲突保护；
- App Server 账号、额度、重置卡和用量读取；
- 安全退出、切换和重启 Codex；
- 重置卡消费 RPC 和幂等参数；
- 本地账号归因、缓存和项目 / 工具 / Skill 统计。

工作台不得再实现第二套账号识别或认证文件操作。Swift 层只负责调用、解码、服务编排和界面状态。

### 工作台账号协调服务

新增一个工作台内的账号协调层，统一处理：

- 启动、每 60 秒、系统唤醒和官方额度通知触发的刷新；
- 最近成功状态与局部错误降级；
- 当前登录账号解析；
- 切换阶段、切换后验证和操作日志；
- 运行状态、通知和自动重置服务的生命周期；
- 防止旧冷备与工作台并行执行账号自动化。

主窗口和菜单栏只观察同一份协调层状态，不分别启动后端调用。

### 冷备兼容

- 迁移期间保留 `Codex Profile Switcher.app` 当前稳定安装产物。
- 工作台检测到冷备进程运行时，不启动自动重置服务，并提示退出冷备。
- 自动重置的 outcome、last-attempt 和 idempotency 状态继续使用兼容旧 bundle 的持久化域，避免冷备切换时重复消费。
- 新工作台验收前不移动或删除旧账号工具源码和安装包。

## 数据流

### 状态刷新

```text
启动 / 60 秒轮询 / 系统唤醒 / account/rateLimits/updated
                         ↓
               status --json
                         ↓
       校验当前账号、额度、重置卡、运行状态
                         ↓
     AccountRuntimeSnapshot（最近成功值 + 新鲜度 + 错误）
            ↓               ↓                ↓
         菜单栏          账号管理页        操作日志
```

刷新失败时保留最近一次成功快照，并明确显示缓存时间与错误；没有可靠值时显示 `--`，不得显示假 `0%`。

### 当前登录账号

“当前登录账号”使用后端 `active_profile` 和 `desktop_status` 的托管结果。历史任务账号推断和统计归因仅进入高级诊断，不得覆盖菜单栏或账号页主状态。

### 账号切换事务

切换必须按以下阶段执行：

1. 记录切换前账号。
2. 退出当前 Codex 进程并等待完成。
3. 保存由 Codex 原子刷新后的最新认证文件，遇到账号冲突立即中止。
4. 接管目标 profile。
5. 重新启动 Codex 并等待完成。
6. 重新读取真实账号状态。
7. 只有 `active_profile` 和托管状态均确认目标账号后，才显示成功并写入日志。
8. 任一步失败都保留阶段和安全回退说明，不伪报成功。

### 自动重置

完整保留现有规则：

- 只有官方 `rateLimitReachedType` 明确报告额度耗尽时才允许消费。
- 到期或即将到期本身绝不触发消费。
- 选择最早到期的可用重置卡。
- 持久化幂等键，失败尝试至少间隔十分钟。
- `reset`、`alreadyRedeemed`、`nothingToReset` 和 `noCredit` 作为终态去重。
- 成功后刷新官方状态、发送通知并写入操作日志。

## 信息架构

工作台侧栏的当前阶段模块：

1. 概览
2. 操作日志
3. 账号管理
4. 项目分析
5. 工具与 Skill

“任务与线程、Hook 与自动化、上下文健康”继续显示为未来能力，但不在本次范围内实现。

## 菜单栏设计

### 状态项

菜单栏状态项只显示：

- 当前登录账号的首要额度百分比；
- Codex 真实工作状态圆点。

悬停说明包含当前账号、额度窗口和状态，例如：`master · 5小时剩余 87% · 运行中`。切换过程中显示稳定的进度状态，切换完成后再更新账号和额度。

### 展开面板

按以下顺序展示：

1. 当前登录账号与 Codex 工作状态。
2. 两档官方额度、重置时间和数据新鲜度。
3. 重置卡数量与最近到期时间。
4. 两个账号的快速切换。
5. 最近两至三条重要操作。
6. 打开工作台、打开 Codex、刷新、设置和退出。

菜单栏不放趋势图、排行、多页导航或逐张重置卡长列表。

## 账号管理页设计

页面阅读顺序：

1. **当前登录账号**：账号名、套餐、Codex 工作状态和托管状态。
2. **官方额度**：两档额度、剩余比例、重置时间、更新时间和缓存状态。
3. **重置卡**：可用数量、每张卡的准确到期时间、提醒状态和自动重置状态。
4. **其他账号**：额度摘要和“切换并重启 Codex”。
5. **账号用量**：今日、7 日、14 日和趋势。
6. **高级诊断**：任务账号推断、统计归因、来源和后端错误，默认折叠。

不再把“最近任务、桌面默认、统计归因”三个技术角色作为页面顶部的同等主卡片。

## 项目分析和工具模块

### 项目分析

- 项目名称与路径；
- 对话数量；
- 本地 token 总量；
- 最近活动时间；
- 数据源和不可用状态。

### 工具与 Skill

- 工具调用排行与最近活动；
- Skill 使用排行与最近时间；
- 统计来源、坏行或缺失数据说明。

两个页面沿用工作台的原生侧栏、连续内容表面、中高信息密度和语义颜色，不复刻旧账号面板的玻璃卡片视觉。

## 启动和进程行为

- 用户手动打开：显示主窗口并建立菜单栏入口。
- 登录 Mac 自动启动：只驻留菜单栏，不主动弹出主窗口。
- 工作台运行时旧 Profile Switcher 应保持退出。
- 退出工作台不退出 Codex。
- 冷备启动前应先退出工作台。

## 错误和状态处理

- `loading`：保留稳定布局，不清空最近成功数据。
- `stale`：显示“暂存”和更新时间，仍允许查看但不伪装为实时。
- `error`：局部错误，不替换整个页面；提供明确重试动作。
- `switching`：显示具体阶段，禁用其他切换动作。
- `verification failed`：显示目标账号与实际账号，不写成功日志。
- `legacy running`：暂停自动重置并提示冷备进程冲突。
- `unknown`：显示 `--` 和原因，不显示假零值或成功绿色。

## Design Lock

继续遵守 `codex-workbench/DESIGN.md` 的 Calm Operations Console：

- 原生 macOS 侧栏和工具栏；
- 工作型中高密度；
- 系统语义色、SF Symbols 和克制卡片；
- 不做营销 hero、彩色图标墙或玻璃拟态大卡片；
- 当前、推断和未知状态必须严格区分；
- 最小 `900×640`、默认 `1160×780`、宽屏 `1440×900`；
- 菜单栏宽约 `380pt`，最大高度约 `520pt`。

## 迁移批次

1. 固定 Profile Switcher `v0.10.1` 冷备基线与产物身份。
2. 同步最新账号后端，扩展工作台数据契约与安装指纹。
3. 迁移运行状态、通知和自动重置服务。
4. 重做菜单栏和账号管理页。
5. 新增项目分析、工具与 Skill 页面。
6. 安装新工作台，退出但不删除旧 App。
7. 完成自动化、真实双向切换、性能和视觉验收。
8. 由用户查看迁移成果并确认。
9. 后续另开旧 App 退役阶段。

## 验收门禁

### 自动化验证

- Profile Switcher Python 完整测试。
- 工作台 Core 测试和 Swift 构建。
- 后端 payload 完整解码契约。
- 当前登录账号语义、缓存降级和切换后验证。
- 通知时间表、自动重置条件、幂等和重试间隔。
- 到期本身不消费的安全回归。
- 操作日志 V1.4 全量回归。
- 安装指纹同时覆盖 Swift 源码、资源和打包的 Python 后端。

### 真实行为验证

- 记录开始时当前登录账号。
- 安装工作台后确认两个账号、两档额度和逐张重置卡详情与旧工具一致。
- 从当前账号切换到另一个账号，再切回并恢复开始时账号；两次都验证 Codex 实际登录状态。
- 菜单栏额度和运行状态随真实状态更新。
- 旧 App 保持安装但不运行；工作台能检测并处理冷备进程冲突。
- 不为了测试触发真实重置卡消费；消费路径通过确定性状态机测试和后端契约验证。

### 视觉验证

- 菜单栏浅色和深色。
- 账号管理页最小、默认和宽屏。
- 项目分析和工具与 Skill 默认窗口。
- 最长账号名、最大数值、缺失值、缓存、错误和切换中状态。
- 运行产物身份、安装路径、二进制 hash、PID 和启动时间均可追溯。
- Visual Verdict 总分不低于 90，任何裁切、越界或错误当前账号语义直接失败。

## 回滚

- 新工作台安装使用原子替换，并保留失败恢复路径。
- 如果账号模块或切换链路失败，退出工作台后可手动启动旧 Profile Switcher 冷备。
- 账号认证文件和 profile 仍由同一成熟后端管理，不进行格式迁移。
- 本阶段不删除冷备，因此回滚不依赖重新登录账号。

## 退役边界

只有全部验收门禁通过、用户查看迁移成果并明确确认后，才进入独立退役阶段。退役阶段将另行设计：停止旧登录项、归档旧原生外壳、删除独立 App、保留必要 CLI 兼容入口和回滚记录。本设计不授权提前执行这些删除动作。
