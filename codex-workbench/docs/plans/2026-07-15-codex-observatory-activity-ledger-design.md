# Codex 观测站全局状态台账设计

## 结论

将“Codex 工具台”命名为“Codex 观测站”，把操作日志升级为可解释的全局状态台账。台账不追踪每个底层工具调用，而是记录被明确建模的状态发生变化：额度与重置次数、项目空间、对话续接、上下文压缩、每日摘要、错误，以及全局规则、Skill、Hook、Plugin 和自动化配置变更。

## 已比较方案

### 方案 A：只做定时轮询

- 优点：实现简单，所有账号走同一条读取路径。
- 缺点：变化发现延迟固定，无法利用官方 `account/rateLimits/updated` 推送。

### 方案 B：只做官方推送监听

- 优点：当前账号的额度变化可以近实时收到。
- 缺点：App 退出、睡眠、网络中断或通知遗漏时会产生空档；官方推送是稀疏更新，不能直接作为完整快照。

### 方案 C：推送 + 快照补扫 + 本地证据差异（采用）

- 当前桌面账号保持 app-server 连接，监听 `account/rateLimits/updated`。
- 收到通知后立即执行完整 `account/rateLimits/read`，再与上次持久化快照比较。
- 所有配置账号每 60 秒补扫；App 启动、Mac 唤醒和网络恢复时立即补扫。
- App 关闭期间只在下次启动发现前后差异，不伪造准确发生时间。

该方案同时满足及时性、可恢复性和证据诚实性。

## 产品命名与图标

- 产品名：`Codex 观测站`。
- 副标题：`本地运行与状态台账`。
- bundle id 保持 `com.hd2yao.codex-workbench`，避免偏好与登录项迁移。
- 图标：macOS 圆角方形底板，石墨灰到深靛蓝表面；由一条时间轴弧线构成抽象字母 `C`，三个节点分别表达项目、对话和全局状态；无文字，小尺寸仍可辨认。

## 额度判定规则

### 权威字段

- 主要/次要额度：`remainingPercent`、`resetsAt`。
- 重置次数：`rateLimitResetCredits.availableCount` 或规范化后的 `reset_credit_details.available_count`。
- 达限状态：`rateLimitReachedType`。
- 来源：Codex app-server `account/rateLimits/read` 与 `account/rateLimits/updated`。

### 分类

1. **正常额度刷新**
   - 上一次剩余额度低于 99%，新快照恢复到 `>= 99%`；
   - 且观察时间落在上一次 `resetsAt` 前 5 分钟至后 15 分钟，或者 App 离线期间该 `resetsAt` 已经过期；
   - 记录为常规事件，展示原定刷新时间与前后额度。
2. **官方侧额度恢复**
   - 上一次剩余额度低于 99%，新快照恢复到 `>= 99%`；
   - 不在正常刷新窗口，且没有同账号、前后 5 分钟内的本地重置次数消费证据；
   - 记录为重要事件，文案为“检测到官方侧额度恢复”；只确认状态由官方接口返回，不宣称是面向所有用户的补偿。
3. **本地重置次数恢复额度**
   - 能关联 Profile Switcher 的 `reset_credit_consumed` 事件；
   - 记录为关键资源事件，并合并展示额度和重置次数的前后值。
4. **重置次数变化**
   - `3 → 4`：重要事件“官方重置次数增加 1”；原因没有字段时显示“官方未提供原因”。
   - `4 → 3`：重要或关键事件；若能关联本地消费则标明消费，否则写“检测到重置次数减少”。
5. **其他额度字段变化**
   - 剩余额度下降、刷新时间变化、达限状态变化均写入台账；连续使用变化为诊断级，达限和数据源失败为关键级。
   - 完全相同的快照只更新检查时间，不生成事件。

## 事件覆盖范围

| 事件 | 重要性 | 证据 | 作用域 |
|---|---|---|---|
| 报错、认证失效、Hook/Automation 失败、额度耗尽 | 关键 | 官方响应、本地失败状态、ledger | 账号/设备/项目 |
| 官方侧额度恢复、重置次数变化、账号切换 | 重要 | app-server + 快照差异 | 账号级，无特定对话 |
| 新增项目空间 | 重要 | `state_5.sqlite` 首次出现的新 `cwd` | 项目级 |
| 对话接续、fork、handoff | 重要 | 结构化 `source_thread_id`、线程关系 | 来源与目标对话 |
| 每日摘要生成 | 重要 | daily digest 文件及其时间 | 全局或摘要项目 |
| 全局规则、Skill、Hook、Plugin、Automation 变更 | 重要 | 受控路径元数据指纹差异 | 全局工作流 |
| 正常额度刷新、上下文压缩 | 常规 | app-server、context card | 账号/对话 |
| 额度使用百分比变化、轮询恢复、数据源重试 | 诊断 | 快照差异、数据源状态 | 账号/设备 |
| 普通新建对话、鼠标/导航、流式 token、完整提示词/回复 | 不记录 | — | — |

## 证据与状态存储

```text
Codex app-server push ─┐
60 秒快照补扫 ─────────┼─> 状态差异分类 ─> operation ledger
state_5.sqlite ────────┤
context cards / digest ┤
规则与能力文件指纹 ─────┘
```

- 事件仍追加到 `~/.codex/operation-ledger/events.jsonl`。
- 观察基线保存到 `~/.codex/operation-ledger/state/`，只包含脱敏字段、路径、时间与文件指纹，不保存文件正文。
- 首次启用项目/工作流监控时只建立基线，不把历史目录全部伪装为“刚刚新增”。
- 新项目空间以首次发现的新规范化 `cwd` 为证据；没有 Codex 项目创建 API 时，文案使用“发现新项目空间”，置信度解释里说明来源。
- 对话标题与项目路径从 `state_5.sqlite.threads` 读取；标题压缩空白并限制展示长度，完整线程 ID 只在详情中显示。
- 对话接续只识别结构化桥接/关系元数据，不扫描或展示普通提示词正文。

## UI 设计契约

### 任务模型

- 主要任务：一眼判断发生了什么、重要程度、属于哪个项目/对话/账号，以及能否定位回去。
- 主操作：选择事件查看详情，打开目标 Codex 对话。
- 次操作：按重要性、来源和状态筛选。

### 时间轴行

```text
22:23  ●  上下文已压缩                         成功
          PreCompact 已生成摘要卡片
          [项目 tools] [对话 Add 操作时间轴日志]  PreCompact Hook · 已核实
```

- 项目与对话名称是首屏字段，不依赖 hover。
- 账号级事件显示 `[账号 hd-master] [全局事件]`，不虚构项目或对话归属。
- 重要性通过时间轴节点尺寸、左侧细强调色和短标签共同表达，不只靠颜色。
- 关键事件使用红/橙语义色；重要事件使用 accent/橙色；常规事件使用分类色低对比；诊断事件使用中性灰。

### Inspector

阅读顺序固定为：

1. 动作、摘要、时间、重要性、可信度。
2. 项目名称、对话名称、账号与作用域。
3. 完整线程 ID、关系中文说明和“在 Codex 中打开”。
4. 来源链、before/after、证据。

`确定` 改为 `已核实`；并显示解释：

- 已核实：有官方数据或本地记录直接证明状态。
- 根据证据推断：状态变化可确认，但原因或准确时间由规则推断。
- 尚无足够证据：只有线索，不能确认。

### 响应式与验收

- 保持既有 `900×640`、`1160×780`、`1440×900` 几何契约。
- 窄屏 scope chips 可换行，标题单行截断，摘要最多两行。
- 宽屏 inspector 保持 350pt；长标题、长项目名和 UUID 不得挤压状态 chip。
- 浅色、深色和 Reduce Motion 均需真实 App 截图验证。

## 错误处理

- app-server 连接中断：写一次关键/重要数据源事件，指数退避重连，60 秒轮询继续工作。
- 快照部分字段缺失：保留上次字段并标注 stale，不把缺失误判为归零。
- SQLite 被锁或 schema 演进：当前轮跳过元数据补全，日志仍可加载，并显示降级提示。
- 受控工作流文件无法读取：只记录数据源警告，不读取认证目录或未知路径。

## 测试策略

- 额度差异分类使用纯 Core 测试：正常刷新、官方恢复、本地重置、次数增减、相同快照去重、离线补扫。
- 线程目录测试：标题清洗、项目解析、首次基线、新项目、结构化续接关系。
- 工作流指纹测试：新增、修改、删除、首次基线、敏感路径排除。
- UI 源码/可访问性契约测试：项目与对话必显、可信度中文解释、重要性非颜色单一表达。
- 完整构建安装后，用真实 ledger 回放并生成最小/默认/宽屏截图。
